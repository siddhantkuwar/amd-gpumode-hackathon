Iteration B
Date: 2026-04-03

Goal:
- Build an aggressive but honest compute improvement on top of the restored baseline.

Probe 1: native HIP quantize+pack+shuffle kernel
- Idea:
  - replace the Triton quant kernel with a custom HIP kernel compiled via `load_inline()`
  - keep the GEMM path unchanged
- Why it seemed plausible:
  - quantization remains the main honest bottleneck after restoring the clean baseline
  - a native HIP kernel could in principle beat the Triton implementation
- Result:
  - correctness passed
  - performance regressed badly
- Benchmark results:
  - k=512, m=4, n=2880: 13.7 us
  - k=7168, m=16, n=2112: 24.7 us
  - k=512, m=32, n=4096: 13.9 us
  - k=512, m=32, n=2880: 14.1 us
  - k=2048, m=64, n=7168: 17.7 us
  - k=1536, m=256, n=3072: 18.3 us
  - Average: about 17.07 us
- Conclusion:
  - the native kernel was honest but slower than Triton

Probe 2: official AITER `dynamic_mxfp4_quant(..., shuffle=True)`
- Idea:
  - use the runner's own quant API directly instead of the local inline Triton copy
- Why it seemed plausible:
  - maybe the packaged AITER path had some runtime advantage over the inlined copy
- Result:
  - correctness passed
  - performance was slightly worse than the honest baseline
- Benchmark results:
  - k=512, m=4, n=2880: 10.6 us
  - k=7168, m=16, n=2112: 21.1 us
  - k=512, m=32, n=4096: 10.7 us
  - k=512, m=32, n=2880: 11.0 us
  - k=2048, m=64, n=7168: 14.0 us
  - k=1536, m=256, n=3072: 12.6 us
  - Average: about 13.33 us
- Conclusion:
  - no real improvement; reverted

Final decision:
- Keep Iteration A as the current honest best state.
- Do not keep either Iteration B probe.

Takeaways:
- The inline Triton fused quant+shuffle kernel remains the strongest honest quant path we tested.
- Small-GEMM ASM routing still helps the edge cases honestly.
- The path to an honest sub-8 submission likely needs a deeper compute change than simple API swaps or a first-pass native kernel.
