Iteration: 002
Date: 2026-04-01

Summary:
- Investigated why the round-1 Python cleanup did not help.
- Confirmed the real measured path is dominated by quantization + GEMM kernels, not import/helper overhead.
- Tried two quant-path changes first; one was API-incompatible with the runner and one changed numerics enough to fail correctness.
- Landed a working hybrid optimization: use direct ASM GEMM only for very small M (`M < 32`), and keep the original `aiter.gemm_a4w4` path for larger shapes.

Why round 1 did nothing:
- The expensive work is in GPU kernels: `dynamic_mxfp4_quant`, `e8m0_shuffle`, and `gemm_a4w4`.
- Moving imports and deleting tiny Python-side scaffolding does not change which GPU kernel runs.
- The benchmark logs repeatedly showed: `not found tuned config ... will use default config!`
- That means the dominant GEMM path stayed the same before and after round 1.
- Removing `B.contiguous()` and unused shape reads was too small to matter relative to the microsecond-level GPU work.
- Net result: round 1 only changed wrapper overhead, not the actual kernel selection or quantization/GEMM behavior.

Iteration-2 sub-attempts that failed:
- Attempt A: call `dynamic_mxfp4_quant(x, shuffle=True)` directly.
  - Result: failed on the remote runner because that API version does not accept a `shuffle` keyword.
- Attempt B: switch to `aiter.get_triton_quant(QuantType.per_1x32)`.
  - Result: ran, but produced mismatched outputs and failed correctness.
  - Takeaway: the quantization path is version-sensitive and must match the runner's expected format exactly.

Final iteration-2 change kept:
- Keep the original known-correct quantization path:
  - `dynamic_mxfp4_quant(x)`
  - `e8m0_shuffle(...)`
- For `M < 32`, bypass `aiter.gemm_a4w4(...)` and call `aiter.gemm_a4w4_asm(...)` directly with the known small-M kernel:
  - `_ZN5aiter41f4gemm_bf16_per1x32Fp4_BpreShuffle_32x128E`
- For `M >= 32`, fall back to the original `aiter.gemm_a4w4(...)` wrapper.

Why this works:
- The direct ASM path removes `aiter.gemm_a4w4` wrapper/config-dispatch overhead for the tiny-M cases where that overhead matters most.
- Larger-M shapes regressed when forced onto the direct path, so the optimization is only enabled where it helped.
